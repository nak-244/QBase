# 化粧品OEM向け 処方ナレッジRAG構想まとめ

---

# 1. 目的

社内に蓄積されている過去の処方実績（試作概要書）をRAG化し、
営業担当が提案資料作成時に類似処方・参考処方を検索できるようにする。

---

# 2. 想定ユースケース

営業担当が自然文で検索：

* 「W洗顔不要なクレンジング」
* 「肌の油分を取りすぎない処方」
* 「植物エキス中心」
* 「セラミドAP必須」
* 「敏感肌向け」
* 「しっとり系」

など。

---

# 3. このシステムの本質

これは単なるFAQ RAGではない。

本質は：

```text
処方ナレッジ検索エンジン
```

である。

重要なのは：

* 成分知識
* 理論解説

ではなく、

* 過去実績検索
* 類似処方検索
* 提案支援
* 処方傾向整理

である。

---

# 5. 推奨Embeddingモデル

## mxbai-embed-large

用途との相性はかなり良い。

理由：

* 日本語対応
* 意味検索に強い
* 「しっとり」「低脱脂」など曖昧語に比較的強い
* Ollama/OpenWebUIと相性が良い

# 7. 重要な考え方

## RAGの主役はLLMではない

```text
RAG = 知識本体
LLM = 整理・説明エンジン
```

である。

性能を決めるのは：

* Metadata設計
* 構造化
* 検索設計
* 辞書設計

である。

---

# 8. 試作概要書はそのままEmbeddingしない

全文Embeddingのみは危険。

理由：

* ノイズが多い
* 注意事項が混ざる
* 類似検索精度が落ちる
* 薬機文言がEmbeddingを汚す

---

# 10. 重要な検索方式

## ① Metadata検索

例：

```text
セラミドAP必須
```

用途：

* 厳密条件検索

---

## ② Vector検索

例：

```text
洗いすぎない
```

用途：

* 意味検索
* 類似検索

---

## ③ 辞書検索

例：

```text
低脱脂
=
マイルド洗浄
=
油分を取りすぎない
```

用途：

* 表現ゆれ吸収

---

# 11. 理想検索フロー

ユーザー入力：

```text
植物エキス中心
W洗顔不要
乾燥肌向け
セラミドAP必須
```

↓

内部処理：

## STEP1 Filter

```json
{
  "成分.セラミドAP": true,
  "成分.植物エキス": true
}
```

↓

## STEP2 Vector Search

```text
W洗顔不要
乾燥肌
低脱脂
しっとり
```

↓

## STEP3 LLM再ランキング

本当に条件に近い処方を整理。

---

# 12. Query Expansion（超重要）

ユーザー入力：

```text
油分を取りすぎない
```

↓

内部変換：

```text
低脱脂
マイルド洗浄
W洗顔不要
保湿型
しっとり
```

これで検索精度が大きく向上する。
