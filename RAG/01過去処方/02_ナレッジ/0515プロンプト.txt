OpenWEBUIのナレッジ・RAG機能を使用して、下記のことを行いたい。
・社内で過去に処方した内容をナレッジとして登録
・処方に詳しくない人（特に新人営業など）が、「W洗顔不要なクレンジング」「敏感肌向けの化粧水」といったあいまいな表現であっても、それらに近しい過去の実績をピックアップする
・場合によっては、「パラベン、シリコン、合成着色料、エタノール不使用」といった、成分に言及した場合でもそれに合致した過去の実績をピックアップする
→こちらを実行することにより、過去の処方実績を資産として活用できる環境を整えたい。

まずは、下記を検討してもらいたい。
いくつか実施したものの、モデルは8b程度が速度的に理想だった。
・qwen系
・gemma系
この2種類で悩んでいるが、他にある場合はそれも含めて、向いているモデルを教えてほしい。

結論から言うと、今回の用途は「一般的なチャット性能」ではなく、
「日本語での曖昧検索 + RAG検索精度 + 指示追従 + 幻覚抑制」が重要です。

そのため、単純なベンチマーク順位ではなく、

* 日本語の曖昧語をどれだけ意味理解できるか
* 「敏感肌向け」「W洗顔不要」などの業務用語をどれだけ近似検索できるか
* RAG参照時に余計な創作をしないか
* 8B級でも実務速度が出るか

で評価すべきです。

現在の方向性（Qwen系 / Gemma系）はかなり妥当です。
特に、既に hallucination テストをされている点は正しい進め方です。

また、現在のOpenWebUI構成では Ollama + mxbai-embed-large を既に利用されています。
Embedding設計としては悪くありません。
ただし「モデル本体」と「Embeddingモデル」は分けて考える必要があります。

まず、本件のタスクを整理します。

---

# 今回のRAGの本質

実際にやりたいことは：

### ① 曖昧検索

例：

* 「敏感肌向け」
* 「W洗顔不要」
* 「高保湿」
* 「ベタつかない」
* 「毛穴ケア」

これは自然言語意味検索。

---

### ② 条件検索

例：

* パラベン不使用
* シリコン不使用
* エタノールフリー

これは半構造化検索。

---

### ③ 類似処方検索

例：

* 過去のクレンジングに近いもの
* 乳液寄りの化粧水
* 使用感が近い

これはEmbedding品質が重要。

---

# つまり必要なのは

単なるLLMではなく：

| 要素          | 重要度 |
| ----------- | --- |
| Embedding品質 | 最重要 |
| 日本語意味理解     | 最重要 |
| RAG参照忠実性    | 超重要 |
| ハルシネーション耐性  | 超重要 |
| 速度          | 重要  |
| 長文生成能力      | 低め  |

です。

ここを間違えると、
「チャットは賢いが検索が弱い」
モデルを選んでしまいます。

---

# 現時点の推奨順位（2026時点の実務観点）

## 1位：Qwen3 8B / 14B 系

最もバランスが良いです。

特に：

* 日本語
* RAG
* 指示追従
* 構造化出力
* 意味理解
* 幻覚抑制

のバランスが非常に良い。

実際、化粧品RAG用途と相性がかなり良い部類です。

以前より：

* 「勝手な補完」
* 「一般論追加」

が減っています。

特に：

* qwen3:8b
* qwen3:14b

はかなり実務向き。

---

## Qwenの強み

### 曖昧語理解が強い

例えば：

* 「しっとり」
* 「重くない」
* 「敏感肌寄り」
* 「W洗顔不要」

この種の業務曖昧語をかなり拾える。

これは営業用途で重要。

---

### JSON/構造化に強い

将来的に：

```json
{
  "category": "クレンジング",
  "features": [
    "W洗顔不要",
    "低刺激"
  ]
}
```

のようなメタデータ生成もやりやすい。

---

### RAGとの相性が良い

「ナレッジに書いてある内容だけ答えろ」
系が比較的安定。

---

## 弱点

### 若干“賢すぎる”

つまり：

* 行間補完
* 意図推測
* 一般論補足

をしがち。

ただしこれはプロンプトでかなり抑制可能。

---

# 2位：Gemma3 12B / 27B 系

Gemma系は：

* 安全性
* 保守性
* 出力安定性

が強い。

特に hallucination 抑制は優秀。

実際、以前の検証でも：

> 「安全だが補完癖あり」

という評価はかなり正確です。

---

## Gemmaの特徴

### 良い点

* 暴走しにくい
* 過剰創作しにくい
* 文章が安定
* 日本語もかなり改善

---

### 悪い点

### 「検索AI」より「回答AI」寄り

つまり：

* 曖昧検索
* 類似検索
* 業務語検索

がQwenより弱い。

---

### 意味検索がやや浅い

「W洗顔不要」
↓
「クレンジング+洗顔統合型」

みたいな概念ジャンプが弱め。

---

# 実務的にはどうするべきか

かなり明確です。

---

# 推奨構成

## 本命

### LLM

* qwen3:14b

### Embedding

* mxbai-embed-large
  または
* bge-m3

---

# なぜ14Bか

8Bでも動きます。

ただ：

* 化粧品
* 処方
* 使用感
* 曖昧語
* 成分条件

を混ぜ始めると、
8Bは意味解像度が落ちやすい。

14Bからかなり改善します。

---

# ただし重要なのは

実は：

> Embedding品質 ＞ LLM本体

です。

RAGの検索精度は、
LLMよりEmbeddingで決まる部分が大きい。

---

# 現在の mxbai-embed-large はどうか

悪くありません。

ただし：

* 日本語特化ではない
* 化粧品業務特化でもない

ため、
将来的には改善余地があります。

---

# 実はかなり有力なのが

## bge-m3

です。

これは：

* 日本語強い
* 多言語強い
* ハイブリッド検索強い
* dense/sparse両対応
* RAG性能高い

で、業務RAGと非常に相性が良い。

---

# 本件で最も重要な設計

モデル選定以上に重要です。

---

# 「ナレッジの構造化」

これが最重要。

例えば：

| 項目   | 内容     |
| ---- | ------ |
| カテゴリ | クレンジング |
| 剤型   | ジェル    |
| 訴求   | W洗顔不要  |
| 肌対象  | 敏感肌    |
| NG成分 | パラベン   |
| 使用感  | しっとり   |
| 備考   | 低刺激設計  |

このように、
半構造化しておくと検索精度が激変します。

---

# なぜか

Embeddingだけだと：

「エタノールフリー」
と
「アルコール不使用」

が揺れます。

なので：

## 自然文 + メタ情報

の両方が必要。

---

# つまり理想構成は

```text
ユーザー質問
↓
Embedding検索
↓
メタ条件フィルタ
↓
LLM要約
```

です。

単純なベクトル検索だけではない。

---

# OpenWebUI単体の限界

これは重要です。

OpenWebUI標準RAGだけだと：

* metadata filter
* hybrid search
* reranker
* sparse検索

が弱い。

そのため将来的には：

* Qdrant
* Weaviate
* Milvus

あたりへ移行した方が伸びます。

---

# 現段階の現実的推奨

## まずはこれ

### LLM

* qwen3:14b

### Embedding

* bge-m3
  または継続で mxbai

### ナレッジ

* 半構造化

### chunk

* 小さすぎない
* 処方単位

---

# 逆に非推奨

## 「全部LLMに理解させる」

これは失敗しやすい。

RAG品質は：

* データ設計
* metadata
* embedding
* chunk設計

でほぼ決まります。

LLMは最後の整形担当です。

---

# 8Bで成立するか

条件付きで成立します。

ただし：

| 条件                    | 必須度 |
| --------------------- | --- |
| ナレッジ整理                | 必須  |
| metadata              | 必須  |
| embedding改善           | 重要  |
| hallucination抑制prompt | 必須  |

です。

逆にこれが弱いと、
70Bでも壊れます。

---

# 現実的な推奨順位

## 総合

1. Qwen3 14B
2. Gemma3 12B
3. Qwen3 8B
4. Gemma3 27B（重い割に恩恵が限定）
5. Llama系（日本語RAG弱め）

---

# 個人的には

本件は：

> 「賢いAI」

より、

> 「検索を間違えないAI」

を作るべき案件です。

そのため、
LLM競争より：

* ナレッジ構造
* Embedding
* metadata
* RAG設計

に工数を割く方が成功率は高いです。

結論としては、

* 現時点の本命は qwen3:14b
* Embedding改善（特に bge-m3）は検討価値が高い
* OpenWebUI標準RAGだけに依存せず、将来的な metadata/hybrid search 設計を前提にした方がよい

です。
